# Research: контейнерный транспорт (002-container-targets)

Все NEEDS CLARIFICATION из Technical Context закрыты решениями R1–R7.

## R1. Обобщение модели транспорта: `ResolvedTarget` вместо `ResolvedSshTarget`

**Decision**: ввести `enum ResolvedTarget { Ssh(ResolvedSshTarget), Container(ContainerTarget) }`
и поменять сигнатуру `Transport::connect(&mut self, target: &ResolvedTarget, auth: &AuthPolicy)`.
Каждый бэкенд принимает только «свой» вариант; чужой вариант — немедленная
`TransportError::BackendUnavailable` (без тихого fallback, как U2 в 001). Выбор
бэкенда по семейству цели делает фабрика в `xxh-cli` (уже существующая точка выбора
russh/ssh-cli), а не ядро. `AuthPolicy` для контейнерного бэкенда игнорируется —
аутентификация там означает доступ к сокету рантайма (Принцип V).

**Rationale**: сохраняет ОДИН trait и пять его методов без изменения семантики
(exec/upload_stream/open_pty/disconnect идентичны); `Session::establish` и вся логика
поверх не меняются вообще. Alternative «generic trait по типу цели» усложняет
диспетчеризацию (`dyn Transport` используется в CLI) ради нулевой выгоды.

**Alternatives considered**: (а) отдельный trait `ContainerTransport` — нарушает
Принцип III (ядро должно не знать семейство); (б) `target: &dyn Any`-подобная
адресация — теряет типизацию; (в) generic `Transport<T: Target>` — ломает
объектную безопасность и текущий код CLI.

## R2. Контейнерные бэкенды: CLI-обёртка vs API-сокет (bollard)

**Decision**: MVP — один бэкенд `ContainerCliTransport` поверх CLI рантайма
(`docker` / `podman` как параметр, одинаковый набор подкоманд: `exec`, `inspect`).
Прямой API-сокет (`bollard` для docker) — за future feature-флагом `docker-api`,
вне MVP. Дополнительные рантаймы — `nerdctl`/containerd и `kubectl exec` — тоже вне
MVP, за feature-флагами `containerd`, `kubectl`; для них уже заложено место в
`ContainerRuntime` (data-model) и в схеме адресации (R4).

**Rationale**:
- CLI-обёртка переносимее: docker и podman дают идентичный CLI, один код покрывает
  оба рантайма; API-сокеты у них различаются (podman: свой REST + docker-совместимый
  слой с оговорками, rootless-пути сокетов нестабильны между дистрибутивами).
- CLI бесплатно уважает пользовательскую конфигурацию рантайма (`DOCKER_HOST`,
  `docker context`, `podman system connection`) — с сокетом это пришлось бы
  реализовывать самим.
- Ноль новых обязательных зависимостей: `bollard` тянет hyper/http-стек в статический
  бинарь (Принцип II страдает размером), CLI-путь использует только `tokio::process`.
- Прецедент в проекте: `SshCliTransport` — та же модель «обёртка над системным CLI
  как режим совместимости»; здесь она становится основной, потому что CLI рантайма —
  это и есть канонический интерфейс пользователя к его контейнерам.
- Trade-off признан: API-сокет дал бы более точные коды ошибок и resize без сигналов;
  это и есть содержимое будущего `docker-api`-бэкенда за тем же trait.

**Alternatives considered**: bollard-only (отвергнуто: не покрывает podman rootless
без танцев, тяжёлые deps); гибрид «bollard для docker + CLI для podman» в MVP
(отвергнуто: два кода доставки при нулевой продуктовой разнице).

## R3. Наличие POSIX `sh` в контейнере — граница MVP

**Decision**: контейнерный путь MVP требует наличия POSIX `sh` внутри контейнера
(как и SSH-путь требует его на хосте: bootstrap-протокол = `sh -s` со stdin).
BusyBox/Alpine/musl — основной поддерживаемый кейс. Полностью «scratch»-образы без
какого-либо `sh` дают явную ошибку класса «доставка» с понятным сообщением до
каких-либо изменений в контейнере; поддержка scratch (доставка статического
busybox/шелла через `cp`-механизм до bootstrap) — задокументированное расширение
вне MVP.

**Rationale**: паритет с SSH-путём (тот же bootstrap-протокол, FR-005 спеки:
переиспользовать механизм детекции); «крайне ограниченный тулинг» из спеки покрыт —
BusyBox `sh` есть в минимальных образах, а bootstrap уже написан под BusyBox/musl.

**Alternatives considered**: доставка собственного статического sh до bootstrap
через `docker cp` (усложняет MVP вторым каналом доставки; отложено).

## R4. Адресация цели в CLI и конфиге

**Decision**: схемный префикс в той же позиции, где сейчас ssh-хост:
- `xxh docker:app1`, `xxh podman:6f0a…` — явный рантайм;
- `xxh container:app1` — семейство контейнеров, рантайм из конфига/флага
  `--runtime`, иначе авто-выбор в документированном порядке: docker → podman
  (первый доступный);
- без схемы (`xxh myhost`) и с `ssh:`-префиксом — SSH-путь, как раньше
  (обратная совместимость 100%);
- в конфиге per-target секции используют ту же строку адреса; новый глобальный ключ
  `container.runtime = "auto" | "docker" | "podman"` (default `auto`).
Резерв для расширений: `k8s:`/`containerd:` схемы за feature-флагами.

**Rationale**: единообразно (одна позиционная «цель» в CLI), различимо (схема),
не конфликтует с ssh-алиасами (двоеточие в начале схемы, а не `host:port` — порт в
xxh задаётся флагом/конфигом, как сейчас). Имя контейнера, совпадающее в обоих
рантаймах, разрешается детерминированно (порядок выше) или явной схемой — закрывает
edge case из спеки без «слияния» пространств имён.

**Alternatives considered**: отдельная подкоманда `xxh exec <ctr>` (ломает «одна
команда, тот же UX»); флаг `--container` при обычном позиционном аргументе
(два способа сказать одно и то же, хуже для конфига); URI `docker://` (длиннее,
без выгоды).

## R5. PTY/интерактив в exec-модели

**Decision**: интерактивная сессия = локально аллоцированная PTY-пара + дочерний
процесс `docker|podman exec -it <ctr> <shell_cmd>`, посаженный на slave; клиент
проксирует stdio через master. Ресайз: по `SIGWINCH` клиента выставлять `winsize`
на master (`TIOCSWINSZ`) и слать `SIGWINCH` процессу CLI — docker/podman CLI сами
транслируют новый размер в exec-сессию через рантайм. Сигналы: байты (Ctrl-C и пр.)
идут через PTY как обычный ввод — семантика для пользователя та же, что в SSH;
завершение шелла закрывает exec и возвращает его exit-code (docker/podman exec
пробрасывают код команды). Смерть контейнера посреди сессии → EOF на master →
завершение с `TransportError::Channel`. Env для шелла передаётся флагами `-e`
exec-команды (значения не логируются).

**Rationale**: полный контроль над терминалом без прямой работы с API attach;
покрывает C-T5-аналог (resize) на обоих рантаймах одинаково. Отличие от SSH
специфицировано отдельно (contracts/container-runtime-backend.md), результат для
пользователя идентичен.

**Alternatives considered**: наследовать наш собственный tty напрямую exec-процессу
(проще, но теряем контроль/наблюдаемость и единый код с будущим API-бэкендом);
websocket/attach через API (это `docker-api`-фича, не MVP).

## R6. Доставка файлов: `exec -i` поток vs `cp`

**Decision**: доставка — только потоком в stdin exec-команды
(`upload_stream("sh -s …", tar)` → `docker exec -i <ctr> sh -s …`), как в SSH-пути.
`docker cp` не используется в MVP.

**Rationale**: bootstrap-протокол уже потоковый (tar в stdin) — ноль изменений в
`Session`/`bootstrap.sh`; один канал доставки (аналог C-T2); `cp` у podman/docker
имеет расхождения с симлинками и владельцами, а также пишет мимо нашего trap-механизма
очистки.

**Alternatives considered**: `docker cp` для больших payload (возможная оптимизация
позже; сначала измерить — exec-поток тем же путём, что SSH, уже удовлетворяет SC-001).

## R7. Тестовая синергия: testcontainers-rs и дуал-транспортная матрица

**Decision**: расширить общий харнес (`crates/xxh-cli/tests/common/mod.rs`)
фикстурой `ContainerFixture` на testcontainers-rs (уже в dev-deps), поднимающей те же
образы `tests/images/{debian,ubuntu,alpine}` БЕЗ требования sshd для контейнерного
пути (образы его содержат — не мешает). Интеграционные сценарии параметризуются
транспортом: идентичный набор проверок «заход → свой шелл → выход → чистота»
прогоняется через SSH-фикстуру (как сейчас) и через контейнерный транспорт против
одного и того же образа. Контейнерные сценарии добавляют второй ассерт: отчёт
изменений ФС контейнера (`docker diff`) пуст после выхода, список/дайджесты образов
не изменились. Alpine — обязательный кейс контейнерного пути. CI-матрица
`integration.yml` расширяется осью `transport: [ssh, container]`.

**Rationale**: контейнеры уже поднимаются в тестах — контейнерный транспорт
тестируется на той же инфраструктуре без дополнительного sshd; требование
конституции VIII (реальный рантайм + проверка неизменности образа) выполняется
буквально.

**Alternatives considered**: отдельный набор образов для контейнерного пути
(дублирование матрицы без выгоды); моки рантайма (запрещено Принципом VIII).
