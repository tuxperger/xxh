# Contract: контейнерный бэкенд (ContainerCliTransport)

**Crate**: `xxh-transport` | **Principles**: I, III, V, VII

Реализация `Transport` поверх CLI рантайма (`docker` | `podman`; research R2).
Расширенные рантаймы (nerdctl/containerd, kubectl) — за feature-флагами, вне MVP,
но обязаны реализовывать этот же контракт.

## connect / attach

- **C-C1**: Последовательность: (1) CLI рантайма найден; (2) демон/сокет отвечает;
  (3) `inspect <ref>` находит контейнер; (4) контейнер запущен. Каждый шаг падает
  различимой ошибкой (data-model: маппинг на `TransportError`), различая
  «не установлен» / «демон недоступен» / «нет прав» / «не найден» / «остановлен».
- **C-C2**: Аутентификация — только права пользователя на сокет/API рантайма;
  ssh-учётные данные не запрашиваются и не читаются (`AuthPolicy` игнорируется).
- **C-C3**: `connect` укладывается в `connect_timeout_s` (включая зависший демон) и
  не виснет.
- **C-C4**: Уважается пользовательская конфигурация рантайма (`DOCKER_HOST`,
  `docker context`, podman connections) — бэкенд не изобретает свою адресацию демона.

## exec / upload_stream (доставка)

- **C-C5**: `exec(cmd)` ⇒ `<runtime> exec <ref> sh -c '<cmd>'`; `upload_stream`
  ⇒ `<runtime> exec -i <ref> <remote_cmd>` со стримом в stdin. Никаких `cp`,
  bind-mount, volume или записи через слои образа: ЕДИНСТВЕННЫЙ канал данных —
  exec (аналог C-T2).
- **C-C6**: Требование к контейнеру — POSIX `sh` (research R3). Его отсутствие —
  явная ошибка доставки ДО каких-либо изменений в контейнере.
- **C-C7**: Bootstrap-протокол 001 (detect → deliver → trap → sweep) переиспользуется
  байт-в-байт: платформа (ОС/арх/libc, включая musl/BusyBox) определяется ВНУТРИ
  контейнера тем же `bootstrap.sh detect`; при эмуляции (arm64-контейнер на x86_64)
  доставляется сборка под платформу контейнера.

## Интерактивная PTY-сессия (open_pty)

Семантика exec-модели, отличная от SSH, но с тем же результатом (research R5):

- **C-C8**: Локальная PTY-пара; `<runtime> exec -it <ref> [-u user] [-e K=V…] <shell_cmd>`
  на slave; проксирование stdio через master.
- **C-C9**: Resize: `SIGWINCH` клиента ⇒ `TIOCSWINSZ` на master + `SIGWINCH`
  exec-процессу; итоговая ширина/высота видна приложениям в контейнере.
- **C-C10**: Ввод (включая Ctrl-C/Ctrl-Z байтами) доставляется в шелл контейнера;
  локальный терминал в raw-режиме; выход из шелла завершает exec и возвращает его
  exit-code как код сессии.
- **C-C11**: Смерть контейнера/демона посреди сессии ⇒ завершение с
  `TransportError::Channel`, терминал клиента восстанавливается, клиент не виснет;
  очистка выполняется в той мере, в какой цель доступна.

## Zero-footprint в контейнере

- **C-C12**: Всё окружение — в одной tmp-директории внутри контейнера; путь
  выбирается из фактически записываемых мест для exec-пользователя ($HOME, `$TMPDIR`,
  `/tmp` — первый записываемый); ни одного записываемого места — явная ошибка до
  доставки (FR-011 спеки).
- **C-C13**: Образ и слои не модифицируются: бэкенд НЕ вызывает `commit`, `build`,
  `export/import` и не пишет вне tmp-директории. После штатного выхода `diff`
  рантайма по контейнеру не содержит артефактов xxh.
- **C-C14**: Очистка — те же правила, что SSH: trap в bootstrap + reconcile sweep
  при следующем заходе после аварийного разрыва; сохранение окружения между
  сессиями — только по явному флагу.

## Диагностика

- **C-C15**: Verbose показывает выбранный рантайм и ссылку контейнера; пути к
  сокетам, значения `-e`-переменных и любые секреты не логируются ни на одном
  уровне (C-T9).
