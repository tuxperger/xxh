# Contract: дуал-транспортное интеграционное тестирование

**Scope**: `crates/xxh-cli/tests`, `tests/images`, CI | **Principle**: VIII (конституция v1.5.0)

Расширяет contracts/integration-testing.md фичи 001: те же тестовые образы теперь
целятся ДВУМЯ транспортами.

## Матрица

| Ось | Значения |
|-----|----------|
| Образ | debian, ubuntu, **alpine (обязателен для контейнерного пути)** |
| Транспорт | ssh (russh; ssh-cli — smoke), container (docker; podman — smoke при доступности) |
| Арх | x86_64, aarch64 (qemu/binfmt) |

- **C-DT1**: Идентичный набор проверок «заход → свой шелл → выход → чистота»
  выполняется обоими транспортами против ОДНИХ И ТЕХ ЖЕ образов
  `tests/images/{debian,ubuntu,alpine}`; логика сценария общая, параметризована
  транспортом.
- **C-DT2**: Контейнерный путь тестируется через testcontainers-rs (уже в dev-deps):
  фикстура поднимает контейнер из тестового образа и подключается контейнерным
  транспортом НАПРЯМУЮ — sshd в контейнере не требуется и не используется этим путём.
- **C-DT3**: Каждый контейнерный сценарий, помимо ассерта чистоты 001 (`~/.xxh`-корень
  удалён), обязан проверять неизменность образа: отчёт изменений ФС контейнера
  (`<runtime> diff`) после выхода не содержит артефактов xxh, а digest образа не
  изменился.
- **C-DT4**: Сценарии классов ошибок: контейнер не найден; контейнер остановлен;
  рантайм недоступен (подмена PATH) — каждый даёт различимую транспортную ошибку и
  НЕ оставляет частично развёрнутого окружения.
- **C-DT5**: Паритет-сценарий: один образ, один конфиг (шелл + плагины) — SSH-заход и
  контейнерный заход дают одинаковый состав доставленных компонентов.
- **C-DT6**: Аварийный разрыв в контейнере: kill клиентского процесса посреди сессии,
  затем повторный заход — sweep вычищает остатки (паритет с SSH-сценарием 001).
- **C-DT7**: Teardown гарантирован (RAII Drop + final cleanup в CI), включая
  контейнеры, поднятые testcontainers-rs; таймауты на сценарий сохраняются.
- **C-DT8**: SSH-набор 001 продолжает проходить без изменений (SC-006 спеки);
  отсутствие docker/podman на машине скипает только контейнерные сценарии
  (skip = явное сообщение, не failure).

## CI

`integration.yml`: матрица расширяется осью `transport: [ssh, container]`
(alpine × container — обязательная ячейка; podman-smoke — отдельная job при
доступности runner-а).
