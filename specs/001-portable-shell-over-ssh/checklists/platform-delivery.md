# Platform Detection & Delivery Requirements Quality Checklist: Portable Shell Environment over SSH

**Purpose**: Проверка КАЧЕСТВА требований к определению целевой платформы хоста и доставке
правильных сборок шеллов/артефактов (полнота, однозначность, непротиворечивость,
измеримость) — interrogate требования, не реализацию. Статус: **[Покрыто]** — задано и
однозначно, **[Двусмысленно]** — есть, но недоопределено, **[Пробел]** — отсутствует.
**Created**: 2026-07-03
**Feature**: [spec.md](../spec.md) · сверено с [contracts/bootstrap-protocol.md](../contracts/bootstrap-protocol.md), [data-model.md](../data-model.md), [contracts/nix-provider.md](../contracts/nix-provider.md) · Принцип II, VI

## Определение ОС и архитектуры хоста

- [ ] CHK001 Определён ли **способ определения ОС и архитектуры** удалённого хоста? [Completeness, Покрыто, Spec §FR-007 / Contract bootstrap-protocol (uname -s -m)]
- [ ] CHK002 Специфицирован ли **состав определяемого профиля** платформы (os + arch + libc musl/glibc)? [Completeness, Двусмысленно, data-model Platform задаёт libc, но §FR-007 упоминает только «архитектура и ОС» — libc-детекция не поднята в spec]
- [ ] CHK003 Задано ли требование, что определение платформы происходит **до записи любых артефактов** на хост? [Consistency, Покрыто, Contract bootstrap C-B1 / Spec §FR-007]
- [ ] CHK004 Определено ли поведение при **нестандартном/непарсящемся `uname`** (неожиданный вывод)? [Edge Case, Пробел, Gap]
- [ ] CHK005 Задано ли поведение на **минимальном хосте** (что если инструмент определения недоступен/ограничен)? [Edge Case, Двусмысленно, §Assumptions задаёт минимальный POSIX-контракт, но случай отсутствия `uname` не описан]
- [ ] CHK006 Определён ли способ распознавания **libc (musl vs glibc)** и его граничные случаи? [Clarity, Пробел, Gap (в spec §FR — нет; в data-model — enum без правил детекции)]

## Список поддерживаемых целевых платформ и неподдерживаемая

- [ ] CHK007 Специфицирован ли **полный список поддерживаемых платформ** (arch/os, musl/glibc)? [Completeness, Покрыто, Constitution §II (Linux x86_64/aarch64/arm, musl/glibc, macOS, BSD) / Plan Target Platform]
- [ ] CHK008 Поднят ли этот список в **нормативный уровень spec** (а не только конституция/plan)? [Traceability, Двусмысленно, в spec §Assumptions список «уточняется на фазе планирования», нормативного §FR нет]
- [ ] CHK009 Определено ли **поведение при неподдерживаемой платформе** (явная ошибка, без частичного окружения)? [Coverage, Покрыто, Spec §FR-007 / §Edge «Неподдерживаемая архитектура/ОС»]
- [ ] CHK010 Задан ли **класс ошибки** для неподдерживаемой платформы различимо (shell vs transport vs plugin)? [Clarity, Покрыто, data-model (ShellError::Unsupported) / §FR-026]
- [ ] CHK011 Однозначно ли, какие платформы **обязательны для MVP**, а какие — целевые «по мере доступности» (macOS/BSD)? [Clarity, Двусмысленно, Plan/§Assumptions не фиксируют обязательность каждой платформы]

## Контентно-адресуемый кеш на хосте

- [ ] CHK012 Определено ли требование **контентно-адресуемого кеша** на хосте? [Completeness, Покрыто, Spec §FR-013 / §VI / Contract bootstrap]
- [ ] CHK013 Задан ли **по какому хешу** адресуется кеш (алгоритм/гранулярность компонента)? [Clarity, Двусмысленно, blake3 задан в plan/data-model, но не в нормативном §FR spec]
- [ ] CHK014 Определены ли требования к **инвалидации** кеша (когда запись считается устаревшей/подлежит замене)? [Completeness, Двусмысленно, §Edge «Устаревшее сохранённое окружение» описывает рассинхрон, но правила инвалидации записи не заданы]
- [ ] CHK015 Задано ли поведение при **коллизии хешей** или повреждённой записи кеша? [Edge Case, Пробел, Gap]
- [ ] CHK016 Определено ли, что при `Ephemeral` кеш **удаляется**, а при `Keep` — переживает сессию (непротиворечиво)? [Consistency, Покрыто, Spec §FR-005/§FR-012 / data-model HostCache]
- [ ] CHK017 Задана ли **верификация целостности** доставленного/кешированного компонента (сверка хеша)? [Coverage, Двусмысленно, Contract bootstrap C-B4 (идемпотентность по hash), явной верификации целостности содержимого нет]

## Нехватка места и отсутствие шелла в кеше

- [ ] CHK018 Задано ли поведение при **нехватке места** на хосте (откат частичных артефактов, чистота)? [Coverage, Покрыто, Spec §FR-032 / §Edge «Нехватка места»]
- [ ] CHK019 Определён ли **класс ошибки** и сообщение при нехватке места? [Clarity, Двусмысленно, §FR-032 задаёт откат, но класс ошибки (shell/transport) не назван]
- [ ] CHK020 Задано ли поведение при **отсутствии нужного шелла в кеше** (до-доставка или ошибка)? [Coverage, Покрыто, Spec §Edge «Отсутствие нужного шелла в кеше» / §FR-011]
- [ ] CHK021 Однозначно ли разграничено «шелл отсутствует в кеше, но доставляем» vs «шелл недоступен вовсе → ошибка класса shell»? [Clarity, Покрыто, Spec §Edge / §FR-011]
- [ ] CHK022 Определён ли **порог/оценка** свободного места до начала доставки (проверка ёмкости заранее)? [Edge Case, Пробел, Gap]

## Объём передачи при повторном подключении и измеримость трафика

- [ ] CHK023 Специфицировано ли, **что именно передаётся** при повторном подключении (только отсутствующие компоненты)? [Completeness, Покрыто, Spec §FR-013 / Contract bootstrap (diff cache)]
- [ ] CHK024 Определён ли **механизм диффа** (сверка имеющихся хешей на хосте до передачи)? [Clarity, Покрыто, Contract bootstrap-protocol (шаг Diff cache)]
- [ ] CHK025 **Измерима** ли минимизация трафика (задан ли критерий, напр. повторный вход −≥50% времени подготовки)? [Measurability, Покрыто, Spec §SC-004]
- [ ] CHK026 Разграничена ли метрика **время подготовки** vs **объём переданных байт** (что именно измеряется)? [Clarity, Двусмысленно, §SC-004 задаёт время; прямой метрики «байты не перезаливаются» нет]
- [ ] CHK027 Заданы ли требования к **дельтам внутри компонента** или только к пропуску целого компонента? [Coverage, Двусмысленно, §FR-013 «только изменившееся»; research уточняет «компонентная гранулярность, внутрикомпонентные дельты — later»]

## Непротиворечивость доставки: базовые шеллы / git-плагины / ⭐ Nix-артефакты

- [ ] CHK028 Задан ли **единый механизм доставки** (Component по хешу) для всех типов артефактов? [Consistency, Покрыто, data-model Component / Contract bootstrap]
- [ ] CHK029 Непротиворечивы ли требования доставки между **базовым шеллом** и **git-плагинами** (одна модель кеша/диффа)? [Consistency, Покрыто, §FR-013 / data-model Component]
- [ ] CHK030 Непротиворечивы ли требования доставки для ⭐ **Nix-артефактов** с общей моделью (тот же кеш/очистка/дифф)? [Consistency, Покрыто, Spec §FR-035 / Contract nix-provider C-N3/C-N4/C-N5]
- [ ] CHK031 Согласована ли доставка **рантайм-данных** (terminfo/CA/locale ⭐Nix) с общей моделью компонентов? [Coverage, Покрыто, Contract nix-provider C-N2/C-N3]
- [ ] CHK032 Нет ли **конфликта** правил удержания кеша (Keep/Ephemeral) между базой, плагинами и Nix? [Conflict, Двусмысленно, база/Nix заданы; для git-плагинов явная привязка к Keep/Ephemeral не зафиксирована (см. plugins.md CHK030/032)]
- [ ] CHK033 Определён ли **порядок доставки/сборки окружения** (шелл → плагины → рантайм-данные) непротиворечиво? [Consistency, Покрыто, Contract bootstrap-protocol (шаги Assemble/Hooks)]
