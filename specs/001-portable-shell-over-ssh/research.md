# Phase 0 Research: Portable Shell Environment over SSH

**Date**: 2026-07-03 | **Feature**: 001-portable-shell-over-ssh

Все версии перепроверены онлайн 2026-07-03 (russh активно развивается, версии не
lock-step между крейтами семейства).

## R1. Версии russh и tokio (перепроверка обязательна)

**Decision**:
- `russh = "0.62"` (последняя — 0.62.1, опубликована 2026-07-02).
- Работа с ключами/agent — через модуль `russh::keys` (на базе RustCrypto `ssh-key`),
  **не** через устаревший отдельный крейт `russh-keys`.
- `russh-config = "0.58"` (0.58.0, 2026-03-18) — парсинг `~/.ssh/config`.
- `tokio = "1.52"` (последняя стабильная в линии 1.52.x, релизы ежемесячно).

**Rationale**: russh — чистый Rust SSH2 клиент/сервер на tokio, что напрямую даёт
zero-footprint и переносимость (Принцип II) без внешнего `ssh`. Ветка 0.6x — актуальная
и активно поддерживаемая (недавние исправления безопасности, в т.ч. DoS в
keyboard-interactive, и улучшения производительности). Пиним минорную версию `0.62` с
диапазоном по патчам; фиксируем в `Cargo.lock`. russh-config покрывает разбор ssh-конфига,
включая директивы, нужные для ProxyJump/Host-alias.

**Alternatives considered**:
- `thrussh` — предшественник russh, менее активен → отклонён.
- `libssh2`/`ssh2` (FFI на C libssh2) — тянет C-зависимость, ломает musl-static и
  zero-dependency цель → отклонён как основной, допустим только как идея fallback, но
  системный `ssh` проще и надёжнее для fallback.
- Только системный `ssh` — не self-contained, зависит от окружения клиента → годится
  лишь как вторая реализация trait.

**Замечание по стабильности API**: russh 0.6x периодически меняет публичный API между
минорными версиями. Поэтому вся работа с russh инкапсулируется в `xxh-transport`
(Принцип III), а обновление russh не затрагивает остальной код.

## R2. Абстракция транспорта (trait Transport)

**Decision**: Определить `#[async_trait] trait Transport` с методами:
`connect`, `exec` (одноразовая команда), `upload`/`stream` (потоковая передача данных),
`open_pty` (интерактивная PTY-сессия), `disconnect`. Две реализации: `RusshTransport`
(основная) и `SshCliTransport` (обёртка над системным `ssh`). Выбор — через
`config.transport` / флаг `--transport`. На первом этапе допускается начать с
`SshCliTransport`, но сигнатуры проектируются под russh (channels, PTY-modes, agent).

**Rationale**: Прямое следование Принципу III. Граница процесса/канала одинакова для обоих
бэкендов: `exec` → отдельный канал; `open_pty` → канал с request_pty + shell; `upload` →
поток по exec (`cat > file`) или SFTP-подобный канал. Абстракция изолирует нестабильный
API russh.

**Alternatives considered**: единый монолитный клиент без trait — нарушает III и
усложняет тестирование (нельзя подменить бэкенд) → отклонён.

## R3. Bootstrap и доставка на хост

**Decision**: Минимальный POSIX `sh` скрипт (встраивается в клиент через `include_str!`),
который: (1) создаёт `~/.xxh` и `~/.xxh/cache`; (2) принимает поток архива (tar+zstd) по
stdin через `exec` канал; (3) распаковывает окружение в директорию, адресуемую по blake3-
хешу содержимого; (4) устанавливает `trap` на `EXIT INT TERM HUP` для очистки, если не
задан режим сохранения; (5) пишет session-marker (`~/.xxh/sessions/<id>`); (6) запускает
шелл. Определение платформы — `uname -s -m` через `exec` до доставки.

**Rationale**: POSIX `sh` есть на всех целевых хостах (минимальный контракт из спеки/
clarify). Zstd даёт лучший коэффициент/скорость для экономии трафика (Принцип VI). Хеш-
адресация → пропуск уже развёрнутого. `trap` + сверка при следующем заходе → гарантия
очистки даже при аварии (Принципы I, V; edge cases спеки).

**Открытый вопрос доставки zstd на хост**: если на хосте нет `zstd`, используем gzip
(`tar+gzip`, часть минимального контракта) как переносимый дефолт, а zstd — как оптимизацию
при обнаружении. Обнаружение — на этапе platform-detection (`command -v zstd`).

**Alternatives considered**: доставлять готовый бинарь-развёртыватель на хост — лишний
перенос и усложнение; POSIX sh достаточно и максимально переносим. rsync — не гарантирован
на хосте и нарушает «только через SSH-сессию» без отдельного канала.

## R4. Контентно-адресуемый кеш

**Decision**: Хешировать каждый упакованный компонент (шелл, каждый плагин, набор
конфигов) blake3; имя в кеше хоста = `~/.xxh/cache/<blake3>`. Перед доставкой клиент
запрашивает у хоста список имеющихся хешей (одна `exec`-команда `ls cache`), передаёт
только отсутствующие. Дельты на уровне компонентов; внутрикомпонентные дельты —
возможное будущее улучшение.

**Rationale**: blake3 — быстрый, стойкий к коллизиям; компонентная гранулярность
покрывает SC-004 (≥50% ускорение повторного входа) без сложной дельта-механики на v1.

**Alternatives considered**: SHA-256 (медленнее, достаточно, но blake3 быстрее для
больших артефактов); бинарные дельты (bsdiff) — сложнее, отложено.

## R5. Модель изоляции плагинов

**Decision**: Плагины — **декларативные пакеты** (данные + объявленные хуки), а не
загружаемый в процесс код. Хуки жизненного цикла (`pre_connect`, `post_deploy`,
`pre_exit`) выполняются как **отдельные процессы** (скрипты/бинарники) в контролируемом
окружении:
- изоляция по **границе процесса** (subprocess), без `dlopen`/динамической линковки в
  адресное пространство клиента/ядра;
- каждый хук получает ограниченный env (`XXH_*`-переменные, `PATH` внутрь xxh-директории),
  таймаут, перехват stdout/stderr;
- ненулевой код возврата или таймаут → плагин помечается сбойным, ошибка класса
  `PluginError` показывается пользователю, **сессия продолжается** (Принцип IV, FR-019);
- секреты хукам не передаются (Принцип V).

**Rationale**: Процессная изоляция даёт crash-isolation и языковую независимость без
`unsafe`/ABI-связанности in-process плагинов. Идеально ложится на «сбой одного плагина не
роняет сессию» и на различимость ошибок (VII).

**Alternatives considered**:
- In-process динамические библиотеки (cdylib + `libloading`) — ABI-хрупкость, сбой плагина
  = краш процесса, `unsafe` → нарушает изоляцию → отклонено.
- WASM-песочница (wasmtime) — сильная изоляция, но тяжёлый рантайм в статическом клиенте,
  ограничивает хуки, преждевременно для v1 → отложено как возможная будущая модель.

## R6. Манифест, resolution, порядок загрузки, конфликты

**Decision**: Манифест `plugin.toml`: `name`, `version` (semver), `dependencies`
(semver-диапазоны на другие плагины), `targets` (совместимые платформы хоста), `hooks`
(объявленные этапы), `provides` (напр. `shell = "zsh"`). Resolver (`semver` крейт):
строит граф зависимостей, проверяет совместимость диапазонов, **обнаруживает конфликт до
доставки** (FR-021) и сообщает конфликтующие плагины/версии. Порядок загрузки —
топологическая сортировка по зависимостям, затем стабильный детерминированный порядок
(явный `priority`, затем имя). Реестр/кеш плагинов на клиенте — контентно-адресуемый
(`~/.local/share/xxh/plugins/<blake3>`), метаданные включённости — в конфиге.

**Rationale**: Прямое покрытие FR-017/018/021 и Принципа IV (детерминированный порядок,
версионирование).

**Alternatives considered**: без semver, «последний выигрывает» — непредсказуемо,
нарушает IV → отклонено.

## R7. Аутентификация и уважение ssh-конфигурации

**Decision**: Через russh: публичные ключи (`russh::keys`), `ssh-agent` (agent-протокол),
параметры из `~/.ssh/config` (`russh-config`), ProxyJump, `known_hosts` (проверка host key).
Плюс интерактивный ввод — пароль и keyboard-interactive (вкл. 2FA-запросы), с запросом у
пользователя (FR-029a). Введённые секреты не логируются (редакция в tracing, Принцип V).
`SshCliTransport` наследует всё это от системного `ssh` естественным образом.

**Rationale**: Полное покрытие FR-029/FR-029a и Принципа V.

## R8. Наблюдаемость и классы ошибок

**Decision**: `tracing` + `tracing-subscriber`; уровни через `-v/-vv`/`--debug`. Кастомный
слой-редактор скрывает поля, помеченные как секретные (ключи, пароли). Ошибки —
`thiserror`-энумы с явными вариантами: `TransportError`, `ShellError`, `PluginError`
(+ `ConfigError`), маппятся в различимые сообщения и коды выхода (FR-026, Принцип VII).

**Rationale**: Покрывает FR-025..028 и требование различимости.

## R9. ⭐ Nix-источник плагинов (stretch goal)

**Decision**: Общая абстракция провайдеров пакетов — trait **`PackageSource`** (aka
PluginProvider) с реализациями `GitProvider`, `LocalProvider`, ⭐`NixProvider`. Ядро
(`xxh-core`) и публичный `xxh-plugin-api` работают только через этот trait и **не знают
про Nix**. `NixProvider` — модуль `xxh-plugins/sources/nix.rs` под cargo-feature
`nix-source` (по умолчанию может быть выключен). Полный контракт —
[contracts/nix-provider.md](./contracts/nix-provider.md). Ключевые решения:

- **Сборка**: `pkgsStatic` (musl) через `nix build` на **клиенте** (нужен Nix с флейками),
  результат — полностью статический бинарь без ссылок на `/nix/store` в рантайме. Аудит
  артефакта на отсутствие store-зависимостей; иначе `PluginError::NotStatic` (FR-038).
- **Воспроизводимость**: spec (`attr`/`expr`) оборачивается в флейк с pinned `nixpkgs`
  rev; `output_hash` детерминирован по (spec, nixpkgs-rev, target).
- **Кросс-компиляция**: target выбирается по результату platform-detection хоста —
  `linux/x86_64`→`pkgsStatic`, `linux/aarch64`→`pkgsCross.aarch64-multiplatform.pkgsStatic`,
  `linux/arm`→`pkgsCross.armv7l-hf-multiplatform.pkgsStatic`. Клиент x86_64 → хост aarch64
  идёт через `pkgsCross`.
- **Ограничение платформ**: musl-static актуален только для Linux; non-Linux (macOS/BSD) →
  `supports_target() = Unsupported`, диагностируется **до** сборки (FR-039).
- **Рантайм-данные**: terminfo (`TERMINFO`), CA-сертификаты (`SSL_CERT_FILE` из `cacert`),
  locale (`LOCALE_ARCHIVE`) собираются из nix closure в переносимый набор и подключаются
  через env в init удалённого шелла (FR-037).
- **Кеш двусторонний**: клиент — `~/.local/share/xxh/nix-cache/<hash>` (не пересобирать);
  хост — общий контентно-адресуемый `~/.xxh/cache/<blake3>` (не перезаливать) (FR-013, VI).
- **Очистка**: Nix-артефакты на хосте лежат в `~/.xxh` и подчиняются общей модели
  (Ephemeral/Keep); на хосте нет `/nix` (Принципы I, V).
- **Деградация**: нет Nix/флейков на клиенте → `availability() = Unavailable`, провайдер
  отключается, база и git/local работают (FR-040, SC-012).

**Rationale**: Прямое следование Принципу IX и FR-033..040. Общий trait изолирует ядро от
Nix (III-подобная изоляция). Feature-флаг гарантирует, что отсутствие Nix не влияет на
сборку/работу базового инструмента.

**Alternatives considered**: делать Nix обязательным бэкендом доставки шеллов — нарушает
IX (Nix стал бы обязательным) → отклонено. Сборка на хосте — нарушает zero-footprint →
отклонено. Динамические (glibc) nix-сборки с патчем rpath/`patchelf` вместо pkgsStatic —
хрупко, тянет store-замыкание → отклонено в пользу полностью статических артефактов.

## R11. ⭐ Research spike «Nix static plugin provider»

**Тип**: инженерный спайк (обязателен перед реализацией `nix-source`). Цель — до кодинга
подтвердить жизнеспособность подхода на реальных сборках.

**Проверить и зафиксировать (артефакт — заметка spike в этом research)**:
1. **Воспроизводимость `pkgsStatic`** типичных CLI (ripgrep, fd, bat, jq): две сборки на
   pinned nixpkgs дают идентичный `output_hash`.
2. **Размер closure** статического артефакта (бинарь + необходимые рантайм-данные) — оценка
   трафика доставки на хост; целевой ориентир — единицы-десятки МБ на инструмент.
3. **Отсутствие рантайм-зависимостей от `/nix/store`**: `ldd`/`file` показывают статический
   бинарь; строковый аудит — нет путей `/nix/store` в критичных code-path; выявить
   инструменты, которым нужны внешние данные (terminfo/CA/locale) и как их пакетировать.
4. **Кросс-сборка под aarch64** (`pkgsCross.aarch64-multiplatform.pkgsStatic`) с клиента
   x86_64: собирается, артефакт статический, запускается на aarch64-хосте.

**Критерии готовности спайка**: для набора эталонных CLI все 4 пункта зелёные, задокументированы
размеры closure и перечень инструментов, требующих доставки рантайм-данных. Отрицательный
результат по инструменту → он помечается несовместимым с Nix-провайдером с понятным
сообщением (FR-038), базовый инструмент не затрагивается.

**Rationale**: pkgsStatic и кросс-сборки имеют известные острые углы (не все пакеты
собираются статически; часть тянет данные извне бинаря). Спайк снимает эти риски до
инвестиции в полную реализацию.

## R12. Nix dev-среда: тулчейн-оверлей и flake-сборка (Принцип X)

**Decision**:
- **Rust-тулчейн в Nix**: оверлей **`oxalica/rust-overlay`**; версия/компоненты берутся из
  `rust-toolchain.toml` (единый источник истины для Nix и не-Nix путей). Компоненты:
  `clippy`, `rustfmt`; таргеты musl/кросс добавляются через оверлей.
- **Сборка Rust в Nix**: **`crane`** (инкрементальная, с отдельным кешированием слоя
  зависимостей `cargoArtifacts`).
- **flake outputs**: `devShells.default`, `packages.xxh` (+ статические/кросс-варианты
  через `pkgsStatic`/`pkgsCross.<target>.pkgsStatic`), `checks` (build + test + clippy +
  fmt). Активация — `nix develop` + direnv (`.envrc: use flake`).
- **CI**: `nix flake check`/`nix build` (источник истины, Cachix для кеша) **и** отдельный
  `cargo`-job без Nix на Linux/macOS (anti-lock-in). Оба обязательны.
- **Пиннинг**: `flake.lock` фиксирует nixpkgs; обновление — контролируемый PR с полным
  `nix flake check`.

**Rationale**: `rust-overlay` + `rust-toolchain.toml` устраняют рассинхрон версий между
Nix и обычным `rustup`-путём. `crane` даёт быстрые инкрементальные пересборки и нативно
покрывает workspace, checks и musl/кросс-цели, переиспользуя ту же `pkgsCross/pkgsStatic`
инфраструктуру, что и Nix-провайдер плагинов (R9/R11) — без дублирования механизмов.

**Alternatives considered**:
- **fenix** (вместо rust-overlay) — тоже хорош; rust-overlay выбран как более
  распространённый в связке с crane и по `rust-toolchain.toml`. Замена локальна.
- **naersk** (вместо crane) — проще для простых случаев, но у crane точнее контроль над
  `cargoArtifacts`, лучше поддержка checks и кросс/musl-матрицы → crane.
- **Только `cargo` без flake** — теряется воспроизводимость и удобная кросс-статика,
  нарушает Принцип X → отклонено (но `cargo`-путь сохраняется как обязательный anti-lock-in).
- **Nix как единственный путь сборки** — vendor lock-in, закрывает контрибьюторов без Nix
  → запрещено Принципом X.

## R13. ⭐ Декларативные Nix-модули конфигурации (Принцип XI)

**Decision**: Два flake output-модуля — `homeManagerModules.default` (основной, per-user) и
`nixosModules.default` (system-wide) — которые **генерируют** канонический конфиг-файл, а не
являются альтернативным источником конфигурации. Полный контракт —
[contracts/nix-config-module.md](./contracts/nix-config-module.md). Ключевые решения:

- **Единый источник схемы (anti-drift)**: типы `Config` в крейте `xxh-config` — канонические.
  Из них экспортируется JSON Schema (через `schemars`) в `nix/config-schema.json`. Опции
  модулей отражают поля `Config` 1:1; **обязательный round-trip-тест** (модуль → `config.toml`
  → парсер `xxh-config`) гарантирует, что модуль и парсер не расходятся. Схема + round-trip
  вместе устраняют дрейф.
- **Типизированные options → eval-валидация**: некорректная декларация падает на
  `nix build`/eval, а не в рантайме инструмента (FR-047, SC-015).
- **Расположение**: HM пишет через `xdg.configFile."xxh/config.toml"` (образец `programs.*`);
  NixOS — через `environment.etc."xxh/config.toml"` (и/или per-user через HM).
- **Синергия**: плагин, объявленный как nixpkgs-пакет, переиспользует ту же
  `pkgsCross`/`pkgsStatic`-инфраструктуру, что и Nix-провайдер (R9/R11) — без отдельного
  механизма (FR-045).
- **Нет рантайм-Nix**: инструмент всегда читает сгенерированный файл; модуль — генератор
  (FR-042, SC-014).

**Rationale**: Прямая реализация Принципа XI. Генерация схемы из единого источника + round-
trip-тест — самый надёжный способ не дать модулю и формату конфига разъехаться. Типизированные
options дают раннюю (eval-time) диагностику вместо рантайм-ошибок.

**Alternatives considered**:
- **Ручное зеркалирование опций без схемы** — высокий риск дрейфа модуля и парсера →
  отклонено в пользу генерации схемы из `xxh-config` + round-trip.
- **Модуль как альтернативный рантайм-источник конфигурации** (инструмент читает Nix-выражение)
  — нарушает Принцип XI и вводит рантайм-Nix → отклонено.
- **Только NixOS-модуль без HM** — неудобно для персонального инструмента (per-user —
  естественный уровень) → HM сделан основным, NixOS — дополнительным.

## R14. Инфраструктура интеграционного тестирования (Принцип VIII)

**Decision**: Интеграционные тесты гоняются против **реальных `sshd`-контейнеров** по
матрице libc/coreutils; оркестрация — **`testcontainers-rs`**. Полный контракт —
[contracts/integration-testing.md](./contracts/integration-testing.md). Ключевые решения:

- **Матрица образов**: Debian + Ubuntu (glibc + GNU coreutils) и **Alpine** (musl +
  BusyBox) как критичный кейс. Каждый обязательный сценарий — против всех трёх.
- **Минимальные образы**: только `sshd` + непривилегированная учётка без права ставить
  пакеты; аутентификация по ключу (пара генерится в тесте), фиксированный host key и
  известный порт для стабильного `known_hosts`.
- **Оркестрация — `testcontainers-rs`** (основной путь): подъём из Rust с RAII-teardown →
  CI-детерминизм и гарантированная очистка даже при падении. `docker-compose` — только для
  локального ручного прогона.
- **Матрица арх**: x86_64 + aarch64 через **qemu/binfmt**; target-таблица общая с
  Nix-провайдером (R11), механизмы не дублируются.
- **Обязательный ассерт чистоты**: каждый сценарий с ожидаемой очисткой проверяет
  отсутствие `~/.xxh`/артефактов после выхода; «сессия открылась» без этого — не пройден.
- **Anti-lock-in**: те же тесты доступны и через `cargo test` c внешним docker (без Nix),
  и в `flake checks` (где есть docker/qemu).

**Rationale**: Прямая реализация усиленного Принципа VIII (v1.4.0). Реальные разнородные
контейнеры без root ловят BusyBox-vs-GNU расхождения и скрытые зависимости; RAII-teardown
`testcontainers-rs` устраняет «висящие» контейнеры и flaky-состояния в CI.

**Alternatives considered**:
- **Моки SSH** — не проверяют реальный bootstrap/очистку на разных libc → отклонено.
- **Только docker-compose** — недетерминированный жизненный цикл в CI, ручной teardown →
  оставлен лишь для локальной отладки.
- **Один дистрибутив** — не покрывает ось musl/BusyBox, где ломается больше всего →
  запрещено Принципом VIII.
- **Реальные VM/облачные хосты** — дорого и медленно для per-PR gate → контейнеры + qemu.

Все NEEDS CLARIFICATION Technical Context разрешены. Ключевые зафиксированные версии:
