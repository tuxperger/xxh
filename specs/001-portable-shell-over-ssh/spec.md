# Feature Specification: Portable Shell Environment over SSH

**Feature Branch**: `001-portable-shell-over-ssh`

**Created**: 2026-07-03

**Status**: Draft

**Input**: User description: "Построй CLI-инструмент, который переносит личное shell-окружение пользователя на удалённый хост по SSH без постоянной установки на этом хосте."

## Overview

xxh — это CLI-инструмент, который переносит личное shell-окружение пользователя
(шелл, конфиги, алиасы, темы, плагины, автодополнение) на любой удалённый хост по
SSH. Одна команда — и на чужой или эфемерной машине пользователь оказывается в
привычном интерактивном шелле со всем своим окружением. После выхода хост остаётся
чистым: временное окружение автоматически удаляется, ничего постоянно не
устанавливается, права root и доступ в интернет на хосте не требуются.

## Clarifications

### Session 2026-07-03

- Q: Какие шеллы поставляются как first-party в MVP? → A: Только один — zsh (архитектура
  плагинов при этом поддерживает несколько шеллов; остальные добавляются как плагины).
- Q: Что инструмент вправе предполагать присутствующим на хосте для бутстрапа? → A:
  POSIX `sh` и базовые утилиты (cat, mkdir, chmod, tar/gzip); ничего сверх этого и без root.
- Q: Какой объём SSH-аутентификации покрывается? → A: Публичные ключи, ssh-agent,
  параметры из ~/.ssh/config (вкл. ProxyJump) и интерактивный ввод (пароль/keyboard-interactive).
- Q: Как квантифицировать «разумный срок» отказа при недоступном хосте (FR-031)? → A:
  Таймаут соединения по умолчанию ~10 с, настраиваемый флагом/конфигом.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Подключение и привычный шелл на удалённом хосте (Priority: P1)

Пользователь выполняет одну команду с именем хоста (то же имя, что в его
`~/.ssh/config`) и через короткое время попадает в интерактивную сессию своего
шелла на удалённой машине — со своими конфигами, алиасами, темой и плагинами.
После выхода из сессии всё развёрнутое окружение с хоста удаляется автоматически, и
машина возвращается в исходное состояние.

**Why this priority**: Это ядро продукта и минимальный жизнеспособный результат. Без
этого сценария инструмент не имеет ценности. Он же доказывает главный принцип —
zero-footprint.

**Independent Test**: Подключиться к тестовому SSH-хосту одной командой, убедиться,
что открылась интерактивная сессия с пользовательским окружением (например,
кастомный prompt/алиас доступен), выйти и проверить, что на хосте не осталось
артефактов.

**Acceptance Scenarios**:

1. **Given** доступный хост, описанный в `~/.ssh/config`, **When** пользователь
   запускает команду подключения с именем этого хоста, **Then** открывается
   интерактивная сессия его шелла по умолчанию с применёнными конфигами и активными
   плагинами.
2. **Given** активную сессию xxh, **When** пользователь завершает сессию (`exit`
   или закрытие шелла), **Then** временное окружение на хосте удаляется полностью и
   команда завершается с успешным кодом.
3. **Given** хост без прав root, без пакетного менеджера и без доступа в интернет,
   **When** пользователь подключается, **Then** окружение всё равно разворачивается
   и сессия работает.
4. **Given** предыдущую сессию, прерванную аварийно и оставившую артефакты, **When**
   пользователь подключается снова, **Then** инструмент обнаруживает и вычищает
   остатки прошлой сессии перед разворачиванием нового окружения.

### User Story 2 - Выбор шелла и его настройка по умолчанию (Priority: P1)

Пользователь может выбрать, какой из поддерживаемых шеллов использовать в конкретной
сессии, а также задать шелл по умолчанию в своём конфиге, чтобы не указывать его
каждый раз. В MVP first-party поставляется один шелл — **zsh** (он же дефолт);
архитектура плагинов при этом поддерживает несколько шеллов, дополнительные добавляются
как плагины.

**Why this priority**: «Личное окружение» бессмысленно без права выбрать свой шелл;
это неотъемлемая часть основного пути и определяет, что именно разворачивается на
хосте.

**Independent Test**: Задать шелл по умолчанию в конфиге, подключиться без флагов и
убедиться, что открылся именно он; затем подключиться с флагом другого шелла и
убедиться, что открылся указанный.

**Acceptance Scenarios**:

1. **Given** заданный в конфиге шелл по умолчанию, **When** пользователь
   подключается без указания шелла, **Then** используется шелл из конфига.
2. **Given** несколько поддерживаемых шеллов, **When** пользователь указывает
   конкретный шелл флагом при запуске, **Then** используется указанный шелл,
   переопределяя значение из конфига.
3. **Given** запрошенный шелл, который не входит в активное окружение пользователя,
   **When** пользователь подключается, **Then** инструмент выдаёт понятную ошибку с
   указанием, как добавить нужный шелл, и не оставляет частично развёрнутого
   окружения.

---

### User Story 3 - Сохранение окружения между сессиями (Priority: P2)

По умолчанию окружение вычищается при выходе, но пользователь может явным флагом
попросить сохранить его на хосте, чтобы следующее подключение к этому хосту было
быстрее (не нужно снова доставлять и распаковывать окружение).

**Why this priority**: Существенно улучшает опыт при регулярных подключениях к одним
и тем же хостам, но не является обязательным для первого рабочего результата.
Поведение по умолчанию (очистка) важнее и покрыто в US1.

**Independent Test**: Подключиться с флагом сохранения, выйти, убедиться, что
окружение осталось на хосте; подключиться повторно и убедиться, что вход заметно
быстрее за счёт переиспользования сохранённого окружения.

**Acceptance Scenarios**:

1. **Given** активную сессию, запущенную с флагом сохранения окружения, **When**
   пользователь выходит, **Then** окружение остаётся на хосте в отведённой
   директории.
2. **Given** сохранённое ранее окружение на хосте, **When** пользователь
   подключается снова, **Then** инструмент переиспользует совпадающие по содержимому
   части и доставляет только изменившееся, ускоряя вход.
3. **Given** отсутствие флага сохранения, **When** пользователь выходит, **Then**
   окружение удаляется полностью (поведение по умолчанию).

---

### User Story 4 - Управление плагинами локально (Priority: P2)

Пользователь управляет набором своих плагинов и конфигов локально на своей машине:
устанавливает, включает, отключает, обновляет и удаляет плагины через подкоманды
CLI. Плагины ставятся из git-репозитория или из локальной директории. Активные
плагины автоматически едут вместе с пользователем на любой хост.

**Why this priority**: Расширяемость и перенос личных плагинов — ключевая ценность
продукта, но она надстраивается над работающим базовым подключением (US1). Может
разрабатываться и тестироваться отдельно.

**Independent Test**: Установить плагин из локальной директории и из git,
перечислить плагины, отключить один, подключиться к хосту и убедиться, что
включённые плагины применились, а отключённый — нет.

**Acceptance Scenarios**:

1. **Given** доступный git-репозиторий плагина, **When** пользователь устанавливает
   плагин по ссылке, **Then** плагин появляется в списке установленных с именем и
   версией из его манифеста.
2. **Given** локальную директорию с плагином, **When** пользователь устанавливает
   плагин из пути, **Then** плагин добавляется так же, как из git.
3. **Given** установленные плагины, **When** пользователь выполняет команды
   включения/отключения/обновления/удаления, **Then** список активных плагинов
   меняется соответствующим образом и сохраняется в конфиге.
4. **Given** набор включённых плагинов, **When** пользователь подключается к хосту,
   **Then** на хосте применяются только включённые плагины.
5. **Given** плагин с ошибкой (некорректный манифест, сбой на своём этапе), **When**
   разворачивается окружение, **Then** ошибка локализуется этим плагином,
   показывается пользователю, а остальная сессия продолжает работать.

### User Story 5 - Единый конфиг с пер-хостовыми переопределениями (Priority: P3)

Пользователь держит единый конфиг-файл, где задаёт шелл по умолчанию, список
активных плагинов, поведение очистки, транспорт по умолчанию и переопределения для
конкретных хостов. Любую настройку можно переопределить флагом командной строки для
отдельного запуска.

**Why this priority**: Повышает удобство и управляемость при работе с многими
хостами, но базовые сценарии работают и на значениях по умолчанию.

**Independent Test**: Задать глобальные настройки и пер-хостовое переопределение,
подключиться к разным хостам и убедиться, что применяются правильные значения; затем
переопределить настройку флагом и убедиться, что флаг имеет приоритет.

**Acceptance Scenarios**:

1. **Given** конфиг с глобальными настройками, **When** пользователь подключается к
   хосту без пер-хостовых правил, **Then** применяются глобальные значения.
2. **Given** конфиг с пер-хостовым переопределением, **When** пользователь
   подключается к этому хосту, **Then** применяются пер-хостовые значения поверх
   глобальных.
3. **Given** любой источник настройки, **When** пользователь передаёт флаг
   командной строки, **Then** значение флага имеет наивысший приоритет для этого
   запуска.

---

### User Story 6 - Понятный прогресс и различимые ошибки (Priority: P2)

Во время подготовки окружения на хосте пользователь видит понятный прогресс этапов,
а при сбоях — сообщения, по которым ясно, что именно пошло не так и на чьей стороне
(проблема соединения, проблема разворачивания шелла или проблема конкретного
плагина).

**Why this priority**: Диагностика на удалённых машинах дорога; наблюдаемость
напрямую влияет на пригодность инструмента к реальной работе, но требует наличия
базового потока подключения.

**Independent Test**: Смоделировать три класса сбоя (недоступный хост,
неподдерживаемый шелл/платформа, сломанный плагин) и убедиться, что сообщения об
ошибке различимы и подсказывают дальнейшие действия.

**Acceptance Scenarios**:

1. **Given** идущую подготовку окружения, **When** выполняются этапы (соединение →
   определение платформы → доставка/разворачивание шелла → применение плагинов),
   **Then** пользователь видит понятный прогресс по этапам.
2. **Given** сбой на одном из этапов, **When** он происходит, **Then** пользователю
   показывается сообщение, однозначно относящее ошибку к классу «транспорт»,
   «шелл» или «плагин».
3. **Given** запрошенный подробный режим вывода, **When** пользователь его включает,
   **Then** детализация возрастает, при этом секреты и приватные ключи не
   показываются.

### User Story 7 - ⭐ Источник плагинов на основе nixpkgs (Priority: P4 — stretch goal)

> ⭐ **Stretch goal.** Это дополнительная, продвинутая возможность. Базовый инструмент
> и остальные источники плагинов (git, локальный путь) MUST работать полностью без
> неё. Функция включается только когда пользователь явно указывает Nix-плагин и на
> клиенте доступен Nix.

Пользователь указывает в качестве плагина пакет из nixpkgs (по имени атрибута) или
Nix-выражение. Инструмент собирает этот пакет в самодостаточный статический бинарь на
локальной машине пользователя и переносит его в удалённое окружение. На хосте не
требуется ни Nix, ни NixOS, ни root. Так пользователь получает доступ к экосистеме
nixpkgs как к набору переносимых инструментов, ничего не настраивая на хосте.

**Why this priority**: Расширяет экосистему доступных инструментов до всего nixpkgs,
но это надстройка над работающей системой плагинов (US4). Она опциональна, зависит от
наличия Nix на клиенте и не должна влиять на базовые сценарии — поэтому низший
приоритет и пометка stretch.

**Independent Test**: На клиенте с установленным Nix добавить плагин из nixpkgs по
имени атрибута, подключиться к Linux-хосту без Nix/root и убедиться, что инструмент
доступен в удалённом окружении и запускается; затем убедиться, что на клиенте без Nix
этот источник помечается недоступным, а остальной инструмент работает.

**Acceptance Scenarios**:

1. **Given** клиент с доступным Nix, **When** пользователь добавляет плагин, указывая
   пакет nixpkgs по имени атрибута, **Then** пакет собирается локально в
   самодостаточный статический бинарь и становится доступным в удалённом окружении
   после подключения.
2. **Given** клиент с доступным Nix, **When** пользователь задаёт плагин как
   Nix-выражение, **Then** инструмент собирает результат этого выражения тем же
   образом и делает его доступным на хосте.
3. **Given** Linux-хост без Nix, без NixOS и без прав root, **When** переносится
   собранный Nix-плагин, **Then** инструмент работает на хосте, не требуя установки
   Nix и не оставляя постоянного следа (в соответствии с поведением очистки US1/US3).
4. **Given** пакет, которому нужны рантайм-данные помимо бинарника (сертификаты,
   terminfo, локали), **When** плагин переносится на хост, **Then** эти данные тоже
   доставляются и корректно подключаются в удалённом окружении, чтобы инструмент
   работал.
5. **Given** клиент без установленного Nix, **When** пользователь пытается
   использовать Nix-источник, **Then** источник помечается недоступным с понятным
   сообщением, а базовый инструмент и остальные источники плагинов продолжают
   работать без ошибок.

### User Story 8 - ⭐ Декларативная настройка через Nix (Priority: P4 — stretch goal)

> ⭐ **Stretch goal.** Опциональная возможность. Базовый инструмент полностью работает
> без неё: пользователь без Nix настраивает всё через обычный конфиг-файл. Nix-модуль —
> лишь генератор канонического конфига (Принцип XI конституции), наличие Nix в рантайме
> не требуется.

Пользователь NixOS или Home Manager описывает конфигурацию инструмента декларативно в
своём Nix-конфиге вместо ручного редактирования конфиг-файла: включает инструмент,
задаёт шелл по умолчанию, список активных плагинов, пер-хостовые настройки, транспорт и
поведение очистки. Модуль генерирует ровно тот же канонический конфиг-файл, который
читает инструмент. Пользователь без Nix настраивает то же самое вручную и получает
идентичный по возможностям результат.

**Why this priority**: Удобство для Nix-экосистемы (декларативность, воспроизводимость
конфигурации вместе с остальной системой), но это надстройка-генератор над уже
работающим каноническим конфигом (US5). Опциональна и не влияет на базовые сценарии —
низший приоритет, stretch.

**Independent Test**: Описать конфигурацию инструмента в Home-Manager-модуле, применить
её, убедиться, что сгенерирован канонический конфиг-файл с теми же настройками, что и
при ручной правке, и что инструмент читает его без наличия Nix в рантайме.

**Acceptance Scenarios**:

1. **Given** пользователь Home Manager, **When** он декларирует настройки инструмента
   (шелл по умолчанию, активные плагины, пер-хостовые настройки, транспорт, очистку) и
   применяет конфигурацию, **Then** генерируется канонический конфиг-файл на уровне его
   пользователя, и инструмент работает по нему без Nix в рантайме.
2. **Given** пользователь NixOS, **When** он декларирует настройки на уровне системы,
   **Then** канонический конфиг применяется для всех пользователей (или заданных), давая
   тот же результат, что и per-user декларация.
3. **Given** декларация плагина-пакета из nixpkgs в этом же Nix-конфиге, **When**
   конфигурация применена, **Then** этот плагин доступен в удалённом окружении —
   декларация плагина и декларация его источника совпадают (совпадает по смыслу с
   Nix-источником плагинов, US7).
4. **Given** пользователь без Nix, **When** он настраивает инструмент вручную через
   конфиг-файл, **Then** он получает конфигурацию, идентичную по возможностям
   декларативному Nix-пути, и тот же результат.
5. **Given** невалидная декларация в Nix-модуле, **When** пользователь собирает
   Nix-конфигурацию, **Then** ошибка возникает на этапе сборки Nix-конфига (до запуска
   инструмента), а не в рантайме инструмента.

### Edge Cases

- **Недоступный хост**: хост не отвечает, отклоняет соединение или не проходит
  аутентификацию — инструмент завершается с понятной ошибкой класса «транспорт», не
  оставляя артефактов, и не зависает бесконечно.
- **Неподдерживаемая архитектура/ОС хоста**: платформа хоста определена, но для неё
  нет подходящей сборки шелла — инструмент явно сообщает об этом и прекращает работу,
  не разворачивая частичное окружение.
- **Разрыв соединения посреди сессии**: сессия обрывается неожиданно — на хосте могут
  остаться артефакты; при следующем подключении к этому хосту инструмент обнаруживает
  и удаляет остатки перед новым разворачиванием, гарантируя очистку.
- **Нехватка места на хосте**: при доставке/распаковке окружения не хватает места —
  инструмент останавливается с понятной ошибкой, откатывает частично созданные
  артефакты и не оставляет мусора.
- **Отсутствие нужного шелла в кеше хоста**: запрошенный шелл отсутствует в
  сохранённом кеше — инструмент доставляет недостающую часть заново; если доставка
  невозможна (например, нет источника), выдаёт понятную ошибку класса «шелл».
- **Конфликт версий плагинов**: два включённых плагина требуют несовместимых версий
  общей зависимости — инструмент обнаруживает конфликт до разворачивания, показывает,
  какие плагины и версии конфликтуют, и не применяет несовместимый набор молча.
- **Прерывание пользователем (Ctrl-C) во время подготовки**: пользователь прерывает
  процесс — начатое разворачивание откатывается, хост остаётся чистым.
- **Устаревшее сохранённое окружение**: сохранённое окружение не соответствует
  текущему набору плагинов/конфигов пользователя — инструмент доставляет только
  изменившиеся части, не заставляя перекачивать всё заново.

#### ⭐ Граничные случаи Nix-источника (stretch goal)

- **На клиенте не установлен Nix**: Nix-источник недоступен — инструмент показывает
  понятное сообщение и продолжает работать; git- и локальный источники, а также
  базовое подключение не затрагиваются.
- **Архитектура хоста отличается от клиентской**: требуется собрать пакет под целевую
  архитектуру хоста, а не под архитектуру клиента; если целевая сборка невозможна,
  инструмент выдаёт понятную ошибку класса «плагин» и не переносит несовместимый
  бинарь.
- **Хост не Linux**: Nix-источник неприменим на не-Linux хостах — инструмент явно
  сообщает об этом и не пытается развернуть Nix-плагин, не затрагивая остальное
  окружение.
- **Пакету нужны рантайм-данные помимо бинарника** (сертификаты, terminfo, локали):
  эти данные должны быть выявлены, доставлены на хост вместе с бинарём и подключены в
  окружении; иначе инструмент сообщает, что плагин не может работать без них.
- **Пакет не собирается статически**: если пакет нельзя собрать в самодостаточный
  статический артефакт, инструмент отклоняет такой Nix-плагин с понятной ошибкой
  класса «плагин» и не переносит зависимый от хоста динамический бинарь.

#### ⭐ Граничные случаи декларативной настройки через Nix (stretch goal)

- **Конфликт декларативных и ручных настроек**: канонический конфиг сгенерирован
  Nix-модулем, но пользователь также правит файл вручную — источник истины один
  (канонический файл); должно быть однозначно определено и диагностируемо, что имеет
  приоритет (сгенерированный модулем результат при применении Nix-конфигурации) и как
  пользователь узнаёт о расхождении.
- **Невалидная декларация**: некорректные значения в Nix-модуле приводят к ошибке на
  этапе сборки Nix-конфига (до запуска инструмента), а не к сбою в рантайме инструмента.
- **Опции конфига, не выраженные в модуле**: если каноническому конфигу доступна опция,
  которой нет в декларативном модуле, должно быть определено ожидание полноты покрытия
  (модуль стремится покрывать возможности канонического конфига; непокрытые опции
  остаются доступны через ручной конфиг без потери функциональности).
- **Nix отсутствует в рантайме**: сгенерированный конфиг читается инструментом без Nix;
  наличие Nix на машине запуска не требуется.

## Requirements *(mandatory)*

### Functional Requirements

#### Подключение и сессия

- **FR-001**: Инструмент MUST позволять пользователю подключиться к удалённому хосту
  одной командой, принимающей имя хоста, совместимое с записями `~/.ssh/config`.
- **FR-002**: Инструмент MUST открывать интерактивную сессию выбранного шелла на
  удалённом хосте с применёнными пользовательскими конфигами и активными плагинами.
- **FR-003**: Инструмент MUST разворачивать окружение во временной директории на
  хосте (по умолчанию `~/.xxh` или её аналог) без постоянной установки, без
  требования прав root и без пакетного менеджера хоста.
- **FR-004**: Инструмент MUST работать на хостах без доступа в интернет — всё
  необходимое окружение доставляется через установленное SSH-соединение.
- **FR-005**: По умолчанию инструмент MUST полностью удалять развёрнутое окружение с
  хоста при завершении сессии.
- **FR-006**: Инструмент MUST гарантированно вычищать артефакты предыдущих сессий,
  включая оставшиеся после аварийного завершения, не позднее следующего подключения
  к тому же хосту.
- **FR-007**: Инструмент MUST определять платформу удалённого хоста (архитектура и
  ОС) и доставлять подходящую под неё сборку шелла; при неподдерживаемой платформе
  MUST завершаться с явной ошибкой, не оставляя частичного окружения.

#### Шеллы

- **FR-008**: Инструмент MUST поддерживать несколько шеллов (через систему плагинов) и
  позволять пользователю выбрать конкретный шелл для запуска. В MVP first-party
  поставляется один шелл — **zsh**; он же используется как значение по умолчанию, если
  пользователь не задал другое.
- **FR-009**: Инструмент MUST позволять задавать шелл по умолчанию в пользовательском
  конфиге и использовать его, когда шелл не указан явно.
- **FR-010**: Инструмент MUST давать флагу выбора шелла приоритет над значением по
  умолчанию из конфига для конкретного запуска.
- **FR-011**: При запросе шелла, отсутствующего в окружении/кеше, инструмент MUST
  выдавать понятную ошибку класса «шелл» с указанием, как добавить нужный шелл.

#### Сохранение и производительность

- **FR-012**: Инструмент MUST по явному флагу сохранять развёрнутое окружение на
  хосте между сессиями вместо очистки.
- **FR-013**: При повторном подключении с сохранённым окружением инструмент MUST
  переиспользовать совпадающие по содержимому части и доставлять только изменившееся.
- **FR-014**: Инструмент MUST обеспечивать заметно более быстрый повторный вход при
  наличии переиспользуемого сохранённого окружения по сравнению с первым входом.

#### Плагины

- **FR-015**: Пользователь MUST иметь возможность устанавливать, включать, отключать,
  обновлять и удалять плагины через подкоманды CLI.
- **FR-016**: Инструмент MUST поддерживать установку плагинов из git-репозитория и из
  локальной директории.
- **FR-017**: Каждый плагин MUST иметь манифест с именем, версией, зависимостями и
  совместимостью с целевыми платформами хоста; инструмент MUST читать и использовать
  эти данные.
- **FR-018**: Инструмент MUST переносить на хост только включённые плагины и
  применять их в предсказуемом (детерминированном) порядке.
- **FR-019**: Плагины MUST быть изолированы так, чтобы сбой одного плагина не ронял
  подключение целиком; ошибка MUST локализоваться этим плагином и показываться
  пользователю.
- **FR-020**: Плагины MUST иметь возможность подключаться к этапам жизненного цикла
  «до подключения», «после разворачивания» и «перед выходом».
- **FR-021**: Инструмент MUST обнаруживать конфликт версий/зависимостей между
  включёнными плагинами до разворачивания и сообщать пользователю, какие плагины
  конфликтуют, не применяя несовместимый набор молча.

#### Конфигурация

- **FR-022**: Инструмент MUST использовать единый пользовательский конфиг-файл,
  задающий как минимум: шелл по умолчанию, список активных плагинов, поведение
  очистки, транспорт по умолчанию и пер-хостовые переопределения.
- **FR-023**: Инструмент MUST применять пер-хостовые переопределения поверх
  глобальных настроек для соответствующего хоста.
- **FR-024**: Инструмент MUST позволять переопределять любые настройки флагами
  командной строки для конкретного запуска, с приоритетом флага над конфигом.

#### Наблюдаемость и ошибки

- **FR-025**: Инструмент MUST показывать понятный прогресс этапов подготовки
  окружения на хосте.
- **FR-026**: Инструмент MUST делать классы ошибок различимыми для пользователя как
  минимум по трём категориям: «транспорт», «шелл», «плагин».
- **FR-027**: Инструмент MUST поддерживать режим подробного вывода для диагностики.
- **FR-028**: Инструмент MUST никогда не показывать и не логировать секреты и
  приватные ключи, в том числе в подробном режиме.

#### Безопасность и транспорт

- **FR-029**: Инструмент MUST уважать пользовательскую SSH-конфигурацию
  (`~/.ssh/config`, `known_hosts`, ssh-agent, ProxyJump) при подключении.
- **FR-029a**: Инструмент MUST поддерживать методы аутентификации: публичные ключи,
  ssh-agent, параметры из `~/.ssh/config`, а также интерактивный ввод (пароль и
  keyboard-interactive, включая 2FA-запросы) с запросом данных у пользователя. Введённые
  секреты MUST NOT логироваться (см. FR-028).
- **FR-030**: Инструмент MUST передавать все данные только через установленное
  SSH-соединение.
- **FR-031**: При недоступном хосте инструмент MUST завершаться с понятной ошибкой
  класса «транспорт» по истечении таймаута соединения (по умолчанию ~10 с,
  настраиваемого флагом/конфигом), без зависания и без остаточных артефактов на хосте.
- **FR-032**: При нехватке места на хосте или прерывании подготовки (в т.ч. Ctrl-C)
  инструмент MUST откатывать частично созданные артефакты и оставлять хост чистым.

#### ⭐ Источник плагинов на основе nixpkgs (stretch goal — опционально)

> Все требования этого подраздела относятся к дополнительной возможности. Они MUST
> NOT быть предпосылкой для работы базового инструмента или других источников
> плагинов (git, локальный путь).

- **FR-033**: Инструмент SHOULD позволять пользователю указывать плагин как пакет
  nixpkgs (по имени атрибута) или как Nix-выражение.
- **FR-034**: Для Nix-плагинов инструмент MUST выполнять сборку на клиенте и получать
  самодостаточный статический бинарь, переносимый на хост.
- **FR-035**: Инструмент MUST разворачивать перенесённый Nix-плагин на хосте без
  требования Nix, NixOS или прав root на хосте, сохраняя поведение очистки/сохранения
  из US1/US3.
- **FR-036**: При отличии архитектуры хоста от клиентской инструмент MUST собирать
  Nix-плагин под целевую архитектуру хоста; при невозможности — выдавать понятную
  ошибку класса «плагин» и не переносить несовместимый бинарь.
- **FR-037**: Инструмент MUST выявлять необходимые пакету рантайм-данные (например,
  сертификаты, terminfo, локали), доставлять их на хост вместе с бинарём и подключать
  в удалённом окружении.
- **FR-038**: Если пакет не собирается в самодостаточный статический артефакт,
  инструмент MUST отклонять такой Nix-плагин с понятной ошибкой класса «плагин».
- **FR-039**: На не-Linux хостах инструмент MUST считать Nix-источник неприменимым и
  явно сообщать об этом, не затрагивая остальное окружение.
- **FR-040**: При отсутствии Nix на клиенте инструмент MUST помечать Nix-источник
  недоступным с понятным сообщением и продолжать работу базового инструмента и
  остальных источников плагинов без ошибок.

#### ⭐ Декларативная настройка через Nix (stretch goal — опционально)

> Требования этого подраздела относятся к дополнительной возможности. Они MUST NOT быть
> предпосылкой для работы базового инструмента: пользователь без Nix настраивает всё
> через канонический конфиг-файл.

- **FR-041**: Инструмент MUST иметь ОДИН канонический формат конфигурации, и рантайм
  MUST читать только его (плюс переопределения флагами CLI); альтернативные источники
  конфигурации в рантайме запрещены.
- **FR-042**: Декларативный Nix-модуль (Home Manager / NixOS) MUST генерировать ровно
  этот канонический конфиг-файл и MUST NOT вводить рантайм-зависимость от Nix.
- **FR-043**: Декларативный модуль SHOULD позволять задавать как минимум: включение
  инструмента, шелл по умолчанию, список активных плагинов, пер-хостовые настройки,
  транспорт и поведение очистки.
- **FR-044**: Home-Manager-декларация MUST применяться на уровне пользователя;
  NixOS-декларация MUST применяться на уровне системы (для всех или заданных
  пользователей), давая тот же результат.
- **FR-045**: Объявление плагина-пакета из nixpkgs в декларативном модуле MUST давать
  тот же результат, что и Nix-источник плагинов (US7): плагин доступен в удалённом
  окружении; декларация плагина и его источника совпадают.
- **FR-046**: Декларативный (Nix) и ручной пути MUST давать идентичную по возможностям
  конфигурацию и один и тот же результат исполнения.
- **FR-047**: Невалидная декларация MUST приводить к ошибке на этапе сборки
  Nix-конфигурации (до запуска инструмента), а не к сбою в рантайме инструмента.
- **FR-048**: При конфликте сгенерированного модулем и вручную изменённого конфига
  источник истины — канонический файл; приоритет и способ диагностики расхождения MUST
  быть определены и понятны пользователю.

### Key Entities *(include if feature involves data)*

- **Host (хост)**: удалённая машина назначения; идентифицируется именем, совместимым
  с `~/.ssh/config`; имеет платформу (архитектура + ОС), определяющую подходящую
  сборку шелла.
- **Environment (окружение)**: набор шелла, конфигов и включённых плагинов,
  разворачиваемый на хосте во временной директории; может быть эфемерным (по
  умолчанию) или сохраняемым.
- **Shell (шелл)**: конкретный интерактивный шелл, поставляемый и разворачиваемый на
  хосте; один из нескольких поддерживаемых; выбирается флагом или конфигом.
- **Plugin (плагин)**: расширение окружения (шеллы, темы, дотфайлы, инструменты,
  хуки жизненного цикла); имеет манифест (имя, версия, зависимости, совместимость с
  платформами); может быть установлен/включён/отключён/обновлён/удалён; ставится из
  источника плагина.
- **Plugin Source (источник плагина)**: способ поставки плагина. Базовые источники —
  git-репозиторий и локальный путь. ⭐ *Stretch goal*: источник на основе nixpkgs —
  пакет по имени атрибута или Nix-выражение, собираемый на клиенте в самодостаточный
  статический артефакт; требует Nix только на клиенте, на хост Nix не переносится.
- **Manifest (манифест)**: описание плагина — имя, версия, зависимости, совместимость
  с целевыми платформами хоста, участие в этапах жизненного цикла.
- **Config (конфиг)**: единый **канонический** пользовательский файл настроек — шелл по
  умолчанию, активные плагины, поведение очистки, транспорт по умолчанию, пер-хостовые
  переопределения. Единственный источник конфигурации, читаемый рантаймом.
- **Declarative Config Module (декларативный модуль)**: ⭐ *Stretch goal*: Nix-модуль
  (Home Manager / NixOS), декларативно описывающий настройки и **генерирующий**
  канонический Config-файл. Опциональный генератор, не альтернативная система; не
  вводит рантайм-зависимости от Nix. Уровень применения: пользователь (Home Manager)
  или система (NixOS).
- **Session (сессия)**: одно подключение от установления соединения до выхода;
  порождает разворачивание окружения и его очистку (или сохранение по флагу).
- **Cache (кеш хоста)**: сохранённые между сессиями части окружения на хосте,
  адресуемые по содержимому, для ускорения повторного входа.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Пользователь попадает в интерактивную сессию своего шелла на новом
  доступном хосте одной командой, без предварительной ручной настройки хоста.
- **SC-002**: После завершения сессии с поведением по умолчанию на хосте не остаётся
  ни одного артефакта, созданного инструментом (проверяемо сравнением состояния
  хоста до и после).
- **SC-003**: Инструмент успешно разворачивает окружение на хосте без прав root, без
  пакетного менеджера и без доступа в интернет в 100% таких тестовых сценариев.
- **SC-004**: Повторный вход с сохранённым окружением заметно быстрее первого входа
  на том же хосте (целевой ориентир — сокращение времени подготовки не менее чем на
  50%), за счёт доставки только изменившегося.
- **SC-005**: При сбое пользователь по сообщению однозначно определяет класс ошибки
  (транспорт / шелл / плагин) без обращения к внешней документации.
- **SC-006**: Сломанный или несовместимый плагин не прерывает сессию целиком: в не
  менее чем 95% случаев сессия остаётся рабочей, а ошибка локализуется этим плагином.
- **SC-007**: После аварийно прерванной сессии следующее подключение к тому же хосту
  оставляет хост чистым (остатки прошлой сессии вычищены).
- **SC-008**: Секреты и приватные ключи не появляются в выводе инструмента ни в
  одном из режимов, включая подробный (проверяемо инспекцией вывода).
- **SC-009**: Пользователь может выбрать шелл и управлять набором плагинов
  (установка/включение/отключение/обновление/удаление) исключительно средствами CLI,
  без ручного редактирования артефактов на хосте.

#### ⭐ Критерии Nix-источника (stretch goal)

- **SC-010**: На клиенте с Nix пользователь делает пакет из nixpkgs доступным в
  удалённом окружении, указав его как плагин, без какой-либо настройки на хосте и без
  Nix/NixOS/root на хосте.
- **SC-011**: Перенесённый Nix-плагин, которому нужны рантайм-данные (сертификаты,
  terminfo, локали), запускается на хосте корректно — в 100% таких тестовых сценариев
  данные доставлены и подключены.
- **SC-012**: На клиенте без Nix попытка использовать Nix-источник даёт понятное
  сообщение о недоступности, при этом базовый инструмент и источники git/локальный
  путь остаются полностью работоспособными (0 регрессий базовых сценариев).

#### ⭐ Критерии декларативной настройки через Nix (stretch goal)

- **SC-013**: Конфигурация, заданная через декларативный Nix-модуль, и та же
  конфигурация, заданная вручную, дают идентичный по возможностям результат —
  инструмент ведёт себя одинаково в обоих случаях (проверяемо сравнением поведения).
- **SC-014**: Сгенерированный модулем канонический конфиг читается инструментом без
  наличия Nix в рантайме в 100% тестовых сценариев.
- **SC-015**: Невалидная декларация обнаруживается на этапе сборки Nix-конфигурации
  (до запуска инструмента) в 100% случаев некорректного ввода — рантайм инструмента при
  этом не затрагивается.

## Assumptions

- Пользователь уже имеет рабочий SSH-доступ к целевым хостам (ключи/агент и
  `~/.ssh/config` настроены на его локальной машине).
- На хосте доступна пользовательская домашняя директория с правом записи и минимальный
  POSIX-бутстрап: `sh` и базовые утилиты (cat, mkdir, chmod, tar/gzip), достаточные для
  приёма и распаковки доставленного окружения. Сверх этого набора инструмент ничего не
  предполагает; права root не требуются.
- «Личная машина» пользователя (клиент), с которой выполняется команда, содержит все
  плагины, конфиги и источники шеллов, необходимые для разворачивания на хосте.
- Целевые платформы хостов охватывают распространённые серверные/десктопные среды
  (Linux разных архитектур, macOS, BSD); конкретная матрица уточняется на фазе
  планирования и согласуется с конституцией проекта.
- Тема, подсветка и автодополнение считаются частью конфигов/плагинов пользователя и
  не выделяются в отдельную подсистему на уровне спецификации.
- Управление плагинами (установка/обновление из git и локального пути) выполняется на
  клиенте и может требовать локального доступа к соответствующим источникам; на хосте
  доступ к git/интернету не предполагается.
- Поведение по умолчанию — очистка окружения при выходе; сохранение включается только
  явным флагом (в соответствии с принципом zero-footprint конституции).
- ⭐ *Stretch goal (Nix-источник)*: Nix присутствует только на клиенте и только для
  плагинов этого типа; на хосте Nix/NixOS не предполагаются. Целевой охват
  Nix-источника — Linux-хосты; на не-Linux хостах он неприменим. Функция целиком
  опциональна — её отсутствие или недоступность не влияет на базовые сценарии и
  остальные источники плагинов. Согласуется с Принципом IX конституции.
- ⭐ *Stretch goal (декларативная настройка через Nix)*: Nix-модуль — опциональный
  генератор канонического конфига (Home Manager / NixOS), а не альтернативная система
  конфигурации; наличие Nix в рантайме инструмента не требуется. Пользователь без Nix
  настраивает всё вручную и получает идентичную по возможностям конфигурацию.
  Согласуется с Принципом XI конституции (конфиг-файл — единственный источник истины).
